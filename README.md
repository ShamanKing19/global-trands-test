# Найденные проблемы в оригинальном коде
1. Фиксированное количество ретраев без увеличения задержки (можно использовать exponential backoff)
2. "Списывание" попытки при отсутствии свободных операторов, вместо "бесплатного" ретрая
3. Отсутствие транзакций и блокировок, из-за чего данные могут остаться в несогласованном состоянии
4. Состояние гонки при выборке свободного оператора. Несколько воркеров могли выбрать одного и того же оператора для разных звонков.
5. Магические значения вместо enum'ов/констант
6. Отсутствие обработки ошибок запроса ко внешней зависимости
7. Недостаточное логирование
8. Лапшекод в одной джобе без зависимостей. Запросы нужно вынести в классы-репозитории. 
Бизнес логику вынести либо в сервисные классы, либо в UseCase'ы. Прокидывать их через DI.
В идеале разбить бы всю эту логику на вызов отдельных методов, поместить их в UseCase и вызывать его в джобе.
То есть вся бизнес-логика лежала бы в UseCase'е, а `ProcessIncomingCallJob` частный случай его запуска.
9. Возможность двойной обработки одного и того же звонка разными воркерами

## Исправленный вариант находится в `refactored.php`
`Ещё можно было бы подумать над прекращением отправки запросов во внешний api, если он точно упал, чтобы не добивать труп (circuit breaker).`

## Критичность проблем по пунктам
- Критические: транзакции, блокировки, race condition, работа со внешним api (пункты 1, 3, 4, 6, 9)
- Важные: ретраи (пункты 2, 7)
- Было бы хорошо сделать: рефакторинг (пункты 5, 8)

## Тесты, которые я бы добавил в первую очередь
1. Сохранение изначального состояния при ошибке/неудаче вызова внешнего api
2. Выбор разных операторов конкурирующими транзакциями
3. Установка корректных состояний сущностям при каждом сценарии джобы:
   - успех
   - неудача при отсутствии операторов
   - неудача при ошибке внешней зависимости
   - неудача при падении бд

## Прямо сейчас не стоит
1. Сильно заморачиваться с ддосом внешней зависимости, если он лежит
2. Глобально рефакторить и выносить вызов внешней зависимости в отдельную джобу
3. Сильно закапываться в потенциальное масштабирование: менять брокер на rabbitMQ/kafka

## Масштабирование
В случае увеличения нагрузки в 10-50 раз основные проблемы, которые нас будут ожидать:
1. Дедлоки из-за блокировок
2. Увеличение длительности выполнения запроса с выбором оператора, т. к. множество воркеров будут блокировать/разблокировать записи
3. При увеличении количества воркеров может закончиться пул соединений к бд или redis'у
4. Внешняя зависимость - может начать отдавать 429, может надолго упасть
5. Redis - единая точка отказа. Нужно либо масштабировать его, либо переходить на брокер предназначенный для такого рода нагрузок (его тоже нужно будет масштабировать).
6. В случае множества воркеров и маленького количества свободных операторов будет огромная конкуренция за них. Увеличение количества воркеров уже не спасёт.
7. Увеличение нагрузки на log-систему, если пишем их синхронно куда-нибудь по сети.
